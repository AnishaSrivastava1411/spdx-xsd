Subj:     XML Schema Development for Software Evidence Archive (security) Information Exchange
1.  Background.  There is a requirement for an XML Schema design to support Software Evidence Archive (security) Information Exchange.   This data model will support software supply-chain verification services that can be readily accredited and employed in multi-level secure enclaves due to the fact that they employ required security protocols and exchange known and trusted data formats.
2.  Tasks
      a.  security Specification Development and Testing
          (1)  Identify and Document security Functional Objectives
          (2)  Identify and Document  Internal and External Information Exchange requirements
          (3)  Determine Test Driven Development strategy and generate test data
      b.  security XML Development
          (1)  security XML Schema.  Employ the National Information Exchange Model (NIEM) methodology to create a consistent but flexible XML Schema data model that can be applied selectively for various purposes while maintaining validity with respect to a standardized core schema.
          (2)  Information Exchange Package Documentation (IEPD).  Create IEPD XML Schema for:
               (a)  JSON Instance generation and validation (JSON to/from XML + validation)
               (b)  Guard Compatible XML Schema (limited fields + security tags)
               (c)  NATO Conformant security tags (STANAG 4774 tagging)
               (d)  Other purpose specific XML Schema
          (3)  Documentation
               (a)  Add all possible documentation to the Reference XML Schema
               (b)  Create XSLT to auto-generate Documentation for the security specification
      c.  security Specfication Implementation
          (1)  Implementation.  Employ XML Schema aware software development practices to exchange and evaluate security data.
               (a)  XML Validation.  Appropriate for external information exchanges or exchanges across guards
               (b)  Automated Test Generation.  Appropriate for server and client side software verification
          (2)  Tooling.  Create an XML Schema configuration management tool to maintain and improve all schema and automate generation of IEPD implementations with XML Schema, XSLT, Test Instances, and code.
      d.  Messaging Formats
          (1)  NIEM registry.  Post the security XML Schema to NIEM under an existing Domain, or sponsor creation of a Cyber specific domain.
          (2)  MIL STD Inclusion.  Create an Interface Change Proposal for potential sponsorship to create a US and NATO MTF Message for Software Supply Chain Information Exchange Requirements.
      e.  Machine-Readable Workflows
          (1)  Internal
               (a)  Load XML Schema
               (b)  Generate Tests
               (c)  Verify Code
               (d)  Run Code
          (2)  External
               (a)  Load XML Schema
               (b)  Generate Instances
               (c)  Validate Instances
               (d)  Verify Security Tags
               (e)  Transform Instances.  Generate JSON, Binary or other formats for processing and/or delivery
               (f)  Send/Receive Messages
      f.  Data Validation
          (1)  Internal.  XML Schema driven tests
          (2)  External
               (a)  XML Schema.   Validation for content
               (b)  Schematron.   Validation for business rules, and conditional content
      g.  Data Transmission
          (1)  XML Instance
          (2)  Efficient XML
          (3)  JSON
          (4)  Specified Binary Format
      h.  Information Exchange
          (1)  Identify Use Cases
          (2)  Specify formats for Use Cases
          (3)  Create IEPD with all information required for each Use Case
3.  security Information Exchange Specification.  Create an XML Schema defined, unit tested web service that generates, sends, receives, and validates security information.
      a.  XML Schema.  Create an XML Schema for security information
          (1)  Create an XML Schema that is conformant to National Information Exchange Model (NIEM) Reference Schema naming and design rules. (done)
          (2)  Use XSLT to Generate a NIEM  Information Exchange Product Documentation (IEPD) XML Schema  to support the Reference Implementation of security Information Exchange.
          (3)  Use XSLT to Generate a NIEM  Information Exchange Product Documentation (IEPD) XML Schema to support the complete documentation of the security Specification.
          (4)  Use the appinfo annotation components to provide mapping information, business rules, and specification details.
      b.  Configuration Management.  View and edit data items, documentation, and application information for the security XML Schema Specification
          (1)  Use Reference XML Schema to generate a Web View to display security XML Schema documentation and content.
          (2)  Use Reference XML Schema to generate a Web Form to collaboratively edit security XML Schema Content.
          (3)  Use XML SCHEMA to re-generate IEPD XML Schema with changes.
          (4)  Employ security DATA SERVICE and XML VALIDATION to run test builds and provide immediate feedback on the effects of security XML Schema changes.
      c.  User Interface.  View and edit data from a security Implementation.
          (1)  Use XML Schema to generate a Web View to display security data.
          (2)  Use XML Schema to generate a Web Form to enter security data.
          (3)  Employ SYSTEM INTERFACE, security DATA SERVICE and XML VALIDATION to validate submitted and received data.
      d.  System Interface.  Send and Receive security Information
          (1)  Create a service that generates and recieves security JSON data.
          (2)  Create a function that converts JSON data to XML.
          (3)  Create a function that converts XML data to JSON.
          (4)  Create a function that validates XML data, and returns JSON response.
          (5)  Use XML test data to verify functions.
      e.  security Data Service.  Generate and verify security Code and process security data
          (1)  Add all of the names from the original JSON to the "xsd:appinfo" in the IEPD Schema to provide the ability to map directly to the JSON used by existing software.
          (2)  Generate GO Structs / JSON Models for every XML Schema component.
          (3)  Generate XML Instances with sample valid and invalid data.
          (4)  Generate code tests using XML test data.
          (5)  Generate code that generates and consumes security JSON data.
          (6)  Integrate tests into build process.
      f.  XML Validation.  Convert security data to XML and process validation information
          (1)  Integrate a service that performs XML Validation.
          (2)  Parse validation responses to JSON objects for logging and user interfaces.
          (3)  Use XML test data to Generate tests that report validation results for messages and components.
          (4)  Generate code that validates an entire message.
          (5)  Generate code that validates components.
